# Заметки для собеседования: как рассказывать Experiment Lab

## Питч на 60 секунд

Experiment Lab — это воспроизводимый демонстрационный проект по
A/B-тестированию для продуктовой аналитики. Я сделал его, чтобы показать не
только анализ в notebook, а полный workflow: synthetic event log, PostgreSQL
schema, deterministic assignment пользователей в control/treatment, расчет
conversion rate, ARPU, AOV, purchase rate, uplift, p-value и confidence
interval, а затем выдачу результата через FastAPI и Streamlit dashboard.

Demo-эксперимент называется `big_data_checkout_test`. Он моделирует изменение
checkout и отвечает на вопрос: улучшает ли treatment покупательское поведение.
Сильная сторона проекта — воспроизводимость: другой человек может поднять
Docker Compose, подготовить demo data одной командой и увидеть тот же dashboard
с результатами анализа.

## Какие вопросы может задать интервьюер

### Почему deterministic assignment?

Потому что пользователь должен оставаться в одной группе при повторном расчете.
Если один и тот же `user_id` каждый раз попадает в разные группы, метрики будут
меняться из-за assignment noise, а не из-за продукта.

### Почему assignment хранится отдельно от events?

Events описывают поведение пользователя, а assignments описывают exposure к
варианту эксперимента. Разделение позволяет надежно соединять поведение с
группой эксперимента и пересчитывать метрики.

### Почему выбраны эти метрики?

`conversion_rate` показывает, покупают ли пользователи вообще. ARPU показывает
выручку на назначенного пользователя. AOV помогает не потерять средний чек.
`purchase_rate` показывает частоту покупок. Вместе они дают более полный взгляд,
чем одна конверсия.

### Зачем нужен API, если можно посчитать в notebook?

API показывает, как аналитический расчет можно упаковать для dashboard или
демо-продукта. Это полезно для портфолио, потому что демонстрирует не только
расчеты, но и понятную инженерную подачу результата.

## Как объяснить uplift

Uplift — это разница между treatment и control.

```text
absolute uplift = treatment - control
relative uplift = (treatment - control) / control
```

Если conversion в control равна 10%, а в treatment 11%, absolute uplift равен
1 percentage point, а relative uplift равен 10%.

Важно объяснить, что uplift показывает размер эффекта, но сам по себе не
доказывает статистическую надежность результата.

## Как объяснить p-value

`p-value` отвечает на вопрос: насколько вероятно увидеть такую или более
экстремальную разницу, если на самом деле между группами нет эффекта.

Хорошая формулировка на интервью:

```text
Я не трактую p-value как вероятность того, что treatment победил. Я использую
его как меру совместимости наблюдаемой разницы с нулевой гипотезой.
```

## Как объяснить confidence interval

Confidence interval показывает диапазон правдоподобных значений эффекта с
учетом неопределенности оценки.

- Если интервал пересекает ноль, уверенного направления эффекта нет.
- Если интервал полностью выше нуля, это поддерживает положительный эффект.
- Если интервал широкий, выборки может быть мало или данные слишком шумные.

## Какие ограничения честно признать

- Данные synthetic, поэтому проект не доказывает реальный бизнес-эффект.
- Нет SRM check для проверки баланса assignment.
- Нет стратификации пользователей.
- Нет CUPED.
- Нет sequential testing.
- Нет защиты от пересекающихся экспериментов.
- Revenue-метрики могут быть скошенными, поэтому t-test — простая первая
  версия.
- Dashboard сделан для demo и объяснения workflow, а не как BI-система.

## Какие trade-offs обсудить

- Простота против статистической сложности: проект сначала показывает базовые
  методы, которые легко проверить и объяснить.
- Synthetic data против реальных данных: synthetic data делает demo
  воспроизводимым, но ограничивает выводы.
- API/dashboard против notebook-only анализа: упаковка требует больше кода, но
  лучше показывает, как аналитика может быть доступна пользователям.
- Live metrics против saved results: live metrics показывают текущее состояние,
  saved results фиксируют результат конкретного анализа.

## Как проект можно развить

- Добавить SRM check.
- Добавить power analysis и расчет размера выборки.
- Добавить CUPED для снижения дисперсии.
- Добавить стратифицированное разбиение.
- Добавить guardrail metrics.
- Добавить поддержку нескольких одновременных экспериментов.
- Добавить экспорт отчета по эксперименту.
