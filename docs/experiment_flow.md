# Experiment Flow

## Зачем нужен deterministic assignment

Если один и тот же пользователь попадает в разные группы при повторном расчёте, эксперимент становится недостоверным.  
Поэтому в проекте используется deterministic assignment: для пары `experiment_key + user_id` всегда получается один и тот же hash bucket, а значит и одна и та же группа.

Это даёт:

- воспроизводимость;
- предсказуемость;
- возможность безопасно пересчитывать назначения;
- простую проверку логики на тестах.

## Почему assignment хранится отдельно

Назначение в эксперимент — это отдельный факт, не часть профиля пользователя и не часть события.

Отдельная таблица `experiment_assignments` нужна, чтобы:

- зафиксировать, в какую группу попал пользователь;
- защититься от дублей через уникальный ключ `(experiment_id, user_id)`;
- хранить служебные данные назначения, например bucket;
- потом корректно джойнить события к эксперименту при расчёте метрик.

## Жизненный цикл эксперимента

### 1. Создание

Эксперимент создаётся в статусе `draft`.

На этом шаге сохраняются:

- карточка эксперимента в `experiments`;
- варианты в `experiment_variants`;
- как минимум один `control` и один `treatment`.

### 2. Запуск

При запуске:

- сервис проверяет, что эксперимент существует;
- проверяет, что он не находится в статусе `completed`;
- загружает варианты;
- проверяет, что пользователи существуют;
- считает deterministic assignment;
- сохраняет назначения в `experiment_assignments`;
- переводит эксперимент из `draft` в `running`.

### 3. Повторный расчёт

Если для тех же `experiment_key` и `user_id` снова выполнить расчёт, hash bucket будет тем же самым.  
База дополнительно защищает от дублей через `ON CONFLICT DO NOTHING`.

### 4. Завершение

На следующем этапе эксперимент можно будет переводить в `completed`, после чего новые назначения будут запрещены.

## Логика разбиения

Для первой версии используется простой hash-based split:

1. Берём строку `experiment_key:user_id`.
2. Считаем `md5`.
3. Преобразуем hash в bucket от `0` до `<100`.
4. Сопоставляем bucket с накопительными долями вариантов.

Пример:

- `control` — 50%
- `treatment` — 50%

Тогда:

- bucket `0.000000 - 49.999999` идёт в `control`;
- bucket `50.000000 - 99.999999` идёт в `treatment`.

## Почему эта схема подходит для первой версии

- её легко объяснить;
- она воспроизводима;
- она не требует отдельного random service;
- она хорошо тестируется;
- она уже достаточно практична для portfolio-проекта.
