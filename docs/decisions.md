# Decisions

## Зачем нужен synthetic generator

На раннем этапе проекта нет реального event log, но уже нужен поток данных, на котором можно:

- проверять схему БД;
- отлаживать ingestion;
- писать SQL-запросы;
- строить dashboard;
- позже считать A/B-метрики и статистику.

Поэтому в проект добавлен простой synthetic data generator, который создаёт не случайный шум, а понятную продуктовую симуляцию.

## Основная логика генерации

### 1. Пользователи делятся по активности

Каждому пользователю присваивается один из трёх профилей:

- `low`
- `medium`
- `high`

Профиль влияет на:

- долю активных дней;
- число сессий в день;
- вероятность просмотра карточек товара;
- вероятность добавить товар в корзину;
- вероятность покупки;
- вероятность оформить подписку;
- вероятность продления подписки.

Это создаёт реалистичную неоднородность: часть пользователей почти неактивна, часть регулярно возвращается, а небольшая доля выглядит как power users.

### 2. События строятся как воронка

Базовый сценарий внутри сессии такой:

`app_open -> view_item -> add_to_cart -> purchase`

При этом каждый следующий шаг происходит не всегда.  
Именно через эти drop-offs появляется естественная воронка, а не просто равномерный набор событий.

### 3. Revenue-события имеют числовое значение

Для событий:

- `purchase`
- `subscription_start`
- `subscription_renewal`

генерируется `event_value`.  
Это позволяет дальше считать:

- revenue;
- ARPU;
- средний чек;
- подписочную выручку.

### 4. Временная структура сохраняется

У пользователя есть:

- время регистрации;
- набор активных дней;
- несколько сессий в некоторые дни;
- события внутри сессии в логичном порядке по времени.

Это важно, потому что многие аналитические запросы завязаны не только на тип события, но и на его последовательность и timing.

### 5. Подписка связана с более ценным поведением

`subscription_start` появляется только у части пользователей и обычно после успешной покупки.  
`subscription_renewal` генерируется позже с отдельной вероятностью продления.

Такой сценарий проще объяснить, чем полностью абстрактную модель, и он ближе к реальному product analytics use case.

## Почему реализация сделана именно так

- Без тяжёлых моделей и ML.
- С фиксированным random seed для воспроизводимости.
- С явными вероятностями, которые можно быстро объяснить на собеседовании.
- С валидацией записей до загрузки в БД.
- С идемпотентной загрузкой через `event_uuid`.

## Почему это практично для портфолио

Этот этап показывает, что проект умеет не только хранить схему, но и создавать управляемый поток данных для аналитики.

Работодателю это демонстрирует:

- понимание event-driven модели данных;
- умение проектировать synthetic data для аналитических задач;
- работу с PostgreSQL ingestion;
- базовую data validation;
- готовность проекта к следующему этапу: расчёту метрик и анализу экспериментов.
